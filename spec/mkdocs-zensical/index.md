# Rendering: MkDocs and Zensical

This is the source of truth for the **rendering side effects**: how the book's
Markdown becomes a website, what that build generates, and how it deploys.

The book itself is specified one level up, in [../index.md](../index.md). Read
that first. Nothing in this file decides what the book says. The site is a
rendering of prose that already exists, and it is disposable: it can be rebuilt
from the Markdown at any time, and it is never committed. If the site and the
book disagree, the book wins.

## Toolchain

- **[Zensical](https://zensical.org/)** is the static site generator. It is
  configured in `zensical.toml` and renders everything under `docs/` into the
  `site/` directory. `site/` is a build output and is not committed.
- **[uv](https://docs.astral.sh/uv/)** manages the Python toolchain and
  dependencies, declared in `pyproject.toml` and pinned in `uv.lock`.
- **[just](https://github.com/casey/just)** is the command runner. The recipes
  live in the `justfile`.

## Commands

All commands run from the repository root.

- `just` : list all tasks.
- `just build` : build the site into `site/` (`uv run zensical build --clean`).
- `just serve` : serve locally with live reload.
- `just nav` : regenerate the navigation and every other generated artifact
  (see below).
- `just test` : run the validation suite and the cross-reference linking tests.
- `just check` : regenerate navigation, then validate.
- `just emdash` : list any em-dashes left in the Markdown (there should be none).
- `just stats` : print chapter, word, and Wikipedia-link counts.

## What is generated, and what is authored by hand

`tools/gen_nav.py` (run by `just nav`) is the single generator. It reads the
chapter files in `docs/chapters/` and writes, from them:

- the `nav` block in `zensical.toml`, between the `BEGIN GENERATED NAV` and
  `END GENERATED NAV` markers;
- the site home page at `docs/index.md`;
- the table-of-contents page in the front matter;
- the subject index;
- the table of contents in the repository `README.md`;
- the chapter map in `guide_xref/chapters.py`, used by the cross-reference
  extension.

**Never hand-edit a generated region.** Change the chapters, then run `just nav`.
Everything in `zensical.toml` outside the generated `nav` markers is authored by
hand and safe to edit: the site name, description, URLs, theme, and extension
configuration. The generator groups chapters by part, so the navigation always
mirrors the book's structure.

## Navigation

The `nav` block lists every published page grouped by part. Because it is
generated from the chapters on disk, it cannot drift from the book. The
validation suite fails if any page under `docs/` is missing from the navigation,
which keeps orphan pages out of the site.

## Cross-reference linking

Chapters refer to each other by plain-text decimal number ("see chapter 8.1"),
per the house style. The `guide_xref` package at the repository root is a
Markdown extension, enabled in `zensical.toml`, that recognizes those references
at build time and links them to the matching chapter page.

- Its chapter map, `guide_xref/chapters.py`, is generated by `just nav`.
- `tests/test_xref.py` locks in the linking rules, including the version numbers
  and quantities that must never be linked (for example WCAG 2.2, Apache 2.0, an
  OKR graded 1.0).

Authors keep writing plain decimal references; the build does the linking.

## Configuration

`zensical.toml` holds the site configuration under `[project]`: `site_name`,
`site_description`, `site_url`, `repo_url`, `edit_uri`, and copyright. These are
authored by hand. The generated `nav` block sits inside the same file, fenced by
the markers described above.

## Deployment

The GitHub Actions workflow `.github/workflows/docs.yml` runs on every push to
`main`. It installs the locked dependencies with uv, runs the validation suite,
builds the site with Zensical, and deploys the result to GitHub Pages using the
official Pages actions. The published site is at
<https://software-engineering-guide.github.io/software-engineering-guide/>.

For the first deployment, GitHub Pages must be set to deploy from GitHub Actions:
repository **Settings, then Pages, then Build and deployment, then Source,
then GitHub Actions**.

## Rules for this layer

1. **The site is downstream.** Regeneration and building never change the book;
   they only reflect it. The book is correct before `just nav` or `just build`
   ever runs.
2. **Never hand-edit generated files or regions.** Edit the chapters and
   regenerate.
3. **The build is reproducible and disposable.** `site/` and the generated
   navigation can be thrown away and rebuilt from the Markdown at any time.
4. **Tests must pass.** Run `just test` before you consider a rendering change
   done.
