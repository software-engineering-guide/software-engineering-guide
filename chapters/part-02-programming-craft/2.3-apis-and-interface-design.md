# 2.3 APIs and interface design

## Overview and motivation

An API is the contract through which one piece of software offers capability to another. It is where teams, systems, and organizations meet, and it is the most durable and expensive thing to get wrong. Internal function signatures can be refactored freely; a published API is a promise to consumers you may never meet, and breaking it breaks them. As organizations decompose monoliths into services and expose capabilities to partners and the public, the API becomes the primary product surface and the primary integration risk.

For large teams, APIs are the coordination mechanism that allows independent work. Well-designed interfaces let teams change their internals without coordinating with every consumer, which is the whole point of service boundaries. Poorly designed interfaces leak internal detail, force lockstep deployment, and turn a set of services into a distributed monolith. The quality of your API design directly determines how independently your teams can move.

In enterprise and government settings, APIs also carry compliance, security, and longevity obligations. Public-sector APIs may be mandated to follow open standards, remain stable for years, and serve external developers who cannot be coordinated with. Enterprise APIs underpin partner integrations with contractual service levels. This raises the bar on versioning discipline, backward compatibility, governance, and developer experience.

## Key principles

- Design the contract first; the interface is a deliberate product decision, not a byproduct of implementation.
- Optimize for the consumer's experience, not the provider's convenience.
- Backward compatibility is a promise; breaking changes require a new version and a migration path.
- Make the easy thing correct: sensible defaults, predictable errors, consistent conventions.
- Design for failure: idempotency, retries, pagination, and rate limiting are first-class, not afterthoughts.
- Choose the protocol style to fit the interaction, not fashion.
- Govern APIs as products with owners, lifecycles, and documentation.

## Recommendations

### Work API-first and contract-driven

Define and review the API contract — its resources, operations, schemas, and error semantics — before writing the implementation. Use a machine-readable specification so the contract can generate documentation, client and server stubs, mock servers, and validation. This lets consumers begin integration against the mock while the provider builds, and it makes the contract the single source of truth that both sides test against.

### Choose the interaction style deliberately

Select among REST, GraphQL, gRPC, and event-driven messaging based on the interaction, not preference. Use REST for resource-oriented, broadly interoperable, cacheable interfaces. Use GraphQL when diverse clients need flexible, aggregated reads over a rich graph. Use gRPC for high-performance, strongly-typed internal service-to-service calls. Use event-driven messaging for asynchronous, decoupled workflows and for propagating state changes. Many large systems use several styles, each where it fits.

### Version and deprecate with discipline

Adopt an explicit versioning strategy and a published deprecation policy: how changes are classified, how long old versions are supported, and how consumers are notified. Distinguish backward-compatible changes (adding optional fields, new endpoints) from breaking changes (removing or renaming fields, changing types or semantics). Never repurpose an existing field's meaning. Provide overlap windows so consumers can migrate, and communicate timelines well in advance.

### Make error semantics consistent and machine-readable

Return structured, predictable errors with stable machine-readable codes, human-readable messages, and enough context to act, without leaking sensitive internals. Use consistent status semantics across all endpoints so clients can handle errors uniformly. Document every error a consumer might encounter.

### Build in idempotency, pagination, and rate limiting

Make write operations safe to retry by supporting idempotency keys, so a client that retries after a timeout does not double-charge or double-create. Paginate all list endpoints from day one, preferring cursor-based pagination for large or changing datasets. Apply and document rate limits, and return the limit state to clients so they can back off gracefully.

### Govern APIs and invest in developer experience

Treat each API as a product with an owner, a lifecycle, and a catalog entry. Establish design-review or an API-standards board so interfaces are consistent across teams. Invest in developer experience: accurate reference docs, quickstarts, examples, a sandbox, and a changelog. For large ecosystems, a portal or catalog that makes APIs discoverable is essential.

## Trade-offs: pros and cons

| Style | Best for | Pros | Cons |
|---|---|---|---|
| REST / HTTP | Public, resource-oriented APIs | Ubiquitous, cacheable, simple, interoperable | Over/under-fetching; many round trips; loose contracts unless specified |
| GraphQL | Flexible reads for varied clients | Client-specified queries; one endpoint; strong schema | Caching and rate-limiting complexity; query-cost risks; server complexity |
| gRPC | Internal high-performance calls | Fast, compact, strongly typed, streaming | Poor browser support; less human-readable; heavier tooling |
| Event-driven | Async, decoupled workflows | Loose coupling; scalable; resilient | Harder to reason about; eventual consistency; operational complexity |

Versioning strategies trade stability against maintenance: supporting many old versions protects consumers but multiplies the code you must maintain and test. Backward compatibility trades provider freedom for consumer stability — usually the right trade for widely used APIs. The overarching principle is that the cost of a bad API decision is paid by every consumer over the entire life of the interface, so it is worth investing more design effort at the boundary than almost anywhere else.

## Examples

An enterprise payments company exposes a public REST API to thousands of merchants. Every write endpoint accepts an idempotency key so that a network retry never creates a duplicate charge, all list endpoints use cursor pagination, and errors carry stable codes documented in a public reference. A formal deprecation policy guarantees a long support window for any version, with advance notice and migration guides. This discipline is a competitive advantage: integrators trust that the API will not break under them.

A government digital service publishes an open API for citizen data following mandated open standards and an API-first design process. The contract is specified and reviewed before build, published in a central government API catalog, and served with a sandbox so third-party developers — who cannot be individually coordinated — can integrate independently. Long-term backward compatibility is a policy requirement because integrations must survive across administrations and vendor changes, so breaking changes are rare and heavily governed.

## Business case: motivations, ROI, and TCO

Good API design lowers integration cost, which is often the dominant cost in connecting systems and onboarding partners. A clear, stable, well-documented API means consumers integrate in days without support tickets; a poor one generates endless support load, failed integrations, and reputational damage. For APIs that are themselves products, developer experience directly drives adoption and revenue.

The largest hidden cost is breaking changes. Every breaking change forces coordinated migration across all consumers — internal teams and external partners alike — and the total cost scales with the number of consumers and their inability to move in lockstep. Investing in contract-first design, backward compatibility, and versioning discipline up front avoids these expensive, organization-wide migration events. To leadership, frame API quality as the leverage point for team autonomy, partner ecosystem growth, and avoidance of costly forced migrations, and track integration time, support-ticket volume, and breaking-change frequency as evidence.

## Anti-patterns and pitfalls

- **Implementation-first APIs** — the interface leaks internal database structure and changes whenever the implementation does.
- **Silent breaking changes** — repurposing a field or tightening validation without a version bump breaks consumers unpredictably.
- **Chatty interfaces** — designs that require many round trips for one logical operation, hurting performance and usability.
- **Inconsistent conventions** — every endpoint invents its own naming, error format, and pagination, so clients cannot generalize.
- **No pagination or rate limiting** — endpoints that work in testing and collapse under real data volume or load.
- **Non-idempotent writes** — retries cause duplicates; a single network blip corrupts data.
- **Version proliferation** — too many live versions with no deprecation, multiplying maintenance until it is unmanageable.
- **Docs as an afterthought** — undocumented or out-of-date references that push all integration cost onto consumers.

## Maturity model

- **Level 1 — Initial:** APIs emerge from implementation; no shared conventions; breaking changes are common and unannounced.
- **Level 2 — Repeatable:** Teams follow basic REST conventions and version informally, but consistency and docs vary.
- **Level 3 — Defined:** Contract-first design, machine-readable specs, a deprecation policy, and consistent error and pagination conventions across teams.
- **Level 4 — Optimizing:** APIs are governed products in a catalog with strong developer experience, measured adoption, automated compatibility checks, and rare, well-managed breaking changes.

## Ideas for discussion

- How do you decide when an internal API is stable enough to publish externally?
- What is the right support window for deprecated versions in your context, and who pays for it?
- Where should GraphQL or gRPC replace REST internally, and where would they add more complexity than value?
- How do you enforce API consistency across many autonomous teams without becoming a bottleneck?
- How should AI-consumable APIs and agent tool interfaces change your design conventions?
- What automated checks can catch backward-incompatible changes before they ship?

## Key takeaways

- Design the contract first; the API is a product and a long-lived promise.
- Backward compatibility protects consumers; breaking changes need new versions and migration paths.
- Choose REST, GraphQL, gRPC, or events by interaction fit, not fashion.
- Build in idempotency, pagination, rate limiting, and consistent errors from day one.
- Govern APIs as products with owners, catalogs, and strong developer experience.
